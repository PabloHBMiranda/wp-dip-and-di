Como funciona — Contratos

O core define o que pode ser implementado. Nunca como.

Cada estratégia implementa os contratos do core. O tema não sabe qual está ativa.

Resolução sem if/else

// NewsListBlock::render() — o bloco não conhece nenhuma estratégia concreta
public function render(array $attributes): string
{
    $alias = $attributes['strategy']; // ex: "g1"

    // O registry resolve a classe concreta certa — sem nenhum condicional
    $service  = $this->registry->resolve($alias, NewsServiceInterface::class);
    $renderer = $this->registry->resolve($alias, NewsRendererInterface::class);

    return $renderer->render($service->getNews($attributes), $attributes);
}

❌ Com if/else

  • Adicionar estratégia = editar o bloco
  • N estratégias = N condicionais no core
  • Violação do Open/Closed Principle

✅ Com registry + container

  • Adicionar estratégia = novo plugin
  • N estratégias = 0 condicionais no core
  • Core nunca muda para adicionar behavior

Como um plugin se registra

// client-a-plugin.php — ponto de entrada do plugin
add_action('core_theme_register_providers', function (Application $app): void {
    $app->registerProvider(new ClientAServiceProvider());
});

// ClientAServiceProvider.php
public function boot(NewsBlockRegistry $registry): void
{
    $strategy = new StrategyStack('client-a', 'Client A');

    $strategy
        ->bind(NewsRepositoryInterface::class, ClientANewsRepository::class)
        ->bind(NewsRendererInterface::class,   ClientANewsRenderer::class);
    //  ↑ só sobrescreve o que precisa — o resto usa o fallback do "default"

    $registry->register($strategy);
}

O core nunca é tocado. O plugin implementa os contratos e se anuncia pelo hook.

DIP+DI vs Filtros de Bloco

VS

⚡ Filtros ganham quando…

  • Curva de aprendizado é crítica
  • Adoção em tema existente hoje
  • Plugins de terceiros precisam integrar
  • Multisite com blogs isolados
  • Time pequeno sem experiência em DI

🏆 DIP+DI ganha quando…

  • Você mantém N personalizações
  • Contratos são estáveis e documentados
  • Testabilidade é requisito
  • Rastreabilidade em runtime é necessária
  • Você é o dev dos dois lados
Filtros: "você pode se pendurar aqui"
Contratos: "você pode substituir isso aqui"

Desvantagens reais no seu cenário específico

Das 20 possíveis, apenas 3 se aplicam quando você é o dev dos dois lados.

⚡ Cold start

  • Container bootstrapa a cada requisição PHP
  • Custo fixo e pequeno
  • Relevante em alta concorrência

🎯 Granularidade

  • Errar os contratos no início custa refatoração
  • Poucos = inflexível
  • Muitos = complexidade sem retorno

🚀 Deploy

  • Atualizar plugin requer composer dump-autoload na raiz
  • Atrito operacional dependendo do pipeline
  • Resolvível com automação
As outras 17 desvantagens assumiam que clientes iam estender o sistema.
Como você é o desenvolvedor dos dois lados, elas não se aplicam.
